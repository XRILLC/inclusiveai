## Description

This paper proposes an approach to Grammatical Error Correction (GEC) which uses token-level classification instead of the preferred seq2seq approach based on how Neural Machine Translation (NMT) is done. In the seq2seq approach, the model is trained to "translate" an errorful sentence into a corrected one. This approach is slow, requires more training data, and does not offer any explanation for the corrections. Instead, the authors approach the task as a token-level sequence classification task where tokens are labelled with transformations which can then be inputted into a decoder to create the corrected sentence. In comparison to other token-level classification systems, this system does not just have KEEP, DELETE, and REPLACE as transformations, but also has transformations for morphology and grammar (such as changing the case or tense of a word) which the authors claim will increase generalizability. Notably, as many transformations rely on other transoformations, the model is run multiple times on the same sentence with the corrections from previous runs. This allows the model to correct in stages.

In terms of training, the authors first pre-train a model on 9 million synthetically-generated errorful sentences. Then, they fine-tune in two stages using multiple existing GEC training datasets. In order to fine-tune, the existing GEC training datasets needed to be labeled with token-level transformation labels. The authors do this by taking the existing correct and errorful sentence pairs and calculating the token labels which minimize Levenshtein distance (essentially minimum edit distance at the word level).

The first experiment of this paper compared the performance of different architectures using this training process. The authors compared RoBERTa, BERT, XLNet, ALBERT, and GPT-2. An LSTM was used as a baseline. In this experiment, it was found that RoBERTa and XLNet perform better than the other models at this task. Based on the results of this experiment, the authors added two hyperparameters to the model: one which adds a bias to the probability of not changing the token and a second which adds a minimum error probability threshold to the output of the error detection layer. Essentially, these hyperparameters both increase the model's bias toward not changing a token, decreasing recall at the benefit of increased overall performance. The second experiment compared the best performing models to existing NMT models. The GECToR models achieve better performance on the CoNLL-2014 and BEA-2019 datasets than the NMT models at up to 10 times faster inference speed. The best GECToR model achieves an F0.5 score of 66.5 and 73.7 on the CoNLL-2014 and BEA-2019 datasets, respectively. In comparison, the best NMT system achieves an F0.5 score of 65.0 and 73.2 on the same datasets.

I think this comparison is fair as the task is the same and the evaluation is the same. However, it does become difficult to compare between architectures as hyperparameter settings and required input can be very different. The authors of course fine-tune their model's hyperparameters for this specific task, even introducing new additional hyperparameters for the model, but do not give the other models the same attention.

The takeway according to the authors is that a sequence-tagging approach with a pre-trained transformer encoder, token-level transformations beyond simply KEEP, DELETE, and REPLACE, and multi-stage training can result in state-of-the-art results in GEC with fast inference time.

My takeway is that the sequence-tagging approach is a viable alternative to NMT-based approaches in GEC systems, but that it requires more work in designing token-level labels for each language for model to be generalizable. However, this approach has a key advantage of not only generating a corrected sentence, but also a sequence of transformations which can be used to generate an explanation of the error corrections which is extremely useful in the case of computer-assisted second language acquisition.

I would use this system for grammatical error correction, perhaps applying it to another language as in English it is largely outdone by other models. However, as part of an ensemble, it is likely to still be useful. Its key advantage to me is its speed which is much faster than many LLMs would be at this task.

I don't think there are major problems with the approach empirically, but I think there is more that can be explored. For example, the authors find that GPT-2 works very poorly with this training pipeline, hypothesizing that this method does not work well with encoders that were trained as part of an encoder-decoder pipeline. However, I think this should be further explored. I also think that improvement of this model over other sequence classification approaches is a key aspect to explore. This model was able to improve performance using special transformations, so there are many experiments to be done with what transformations are best for generalizability. How many transformation types are too much where the model starts decreasing in performance? Essentially, creating more transformation types is offloading work from the model onto the designer of the token labels. The model is theoretically capable of correcting any sentence using just KEEP, DELETE, and REPLACE, but designing more labels creates a better framework for generalizing corrections. Where do you stop?

## Motivation

As a side project, I have been working on a GEC system for Spanish which uses a fine-tuned BETO (Spanish BERT) model. I was originally working with BARTO (Spanish BART) and framing it as a seq2seq task. I planned to use Wikipedia edit history to create training, but I found grammatical edits were actually quite rare in modern Spanish Wikipedia and the vast majority of minor edits were content corrections, not grammar correction. Therefore, I decided to instead design a system for generating synthetic errors from correct Spanish sentences. While brainstorming this system, I found this paper and I thought the framing of the problem as a token-level classification task was not only very interesting to me, but also created a nice framework for generating synthetic data. I would generate errors by generating gold transformations and working backwards through them to make the errorful sentence. Because I would already have the token-level labels, I decided to try making my system with the same task design and modify the approach of the paper to suit Spanish grammar. This is actually my final project for this class. This was convenient as I could now use BERT-like models with good results which I was already familiar with and the model would run much faster. This paper is not new or necessarily state-of-the-art anymore, but it changed how I was thinking about GEC systems.
