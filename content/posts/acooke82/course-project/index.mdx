## Project description

### Overview & Novelty

This project, titled "LLM Classification Finetuning", is a Kaggle competition which aims to explore how different LLMs react to user inputs and to see if an LLM is able to predict the preferred responses of users. The goal of the competition is to build a model that can accurately predict the preferences of the human judges, determining the likelihood that a test prompt is the ideal choice to present in the future. The project is intended to demonstrate how human feedback can assist with creating better models, leading to a better human experience. There are three sets of data provided to competition participants to assist them with this goal: a training set, a test set, and a sample submission. To build the training dataset, users chatted with two anonymous LLMs and selected which answer they preferred from the two, creating a dataset of user prompts, LLM responses, and user judgements. The test dataset is provided to participants as an example of prompt/response pairs their model can expect and is replaced with alarger, hidden test data set to score the actual submissions. The sample submission serves as an example of how the participant should organize their final csv for it to be accepted on the leaderboard. The accuracy of submissions is based on the log loss between the predicted probabilites and the ground truth values. This project is unique since it is a "Getting Started" Kaggle competition, which are geared towards participants with little machine learning experience who want to get comfortable on the Kaggle platform and data science. The project also provides users with a "Starter Notebook" to give them an idea of what to look for and how to do exploratory data analysis. The project also runs locally in a Kaggle-hosted notebook service and submissions are graded by running a hidden version of the participant's local notebook.

At that time, the task was to build a classifier which looked at a csv containing three different columns and then made predictions on a new dataset based off of the features it had been trained on. ## Future improvements

### Limitations

Although this project was geared towards individuals who are new to data science and provided good resources for learning, there were still limitations present. I think that one limitation for this project was not being able to view more of the test data. There is a possibility that data present in the testing set contained anomalies that were not present in the training set. Identifying and accounting for these could be beneficial to the model's accuracy. Additionally, having to use and submit through the Kaggle notebook environment could be another limitation. Though the notebook system is potentially more user-friendly to new participants, users risk losing their work by hosting it through the website and may be uncomfortable with having their code publically viewable to all users. Users might also be tempted to view other participant's submissions, limiting their own creativity and abilities.      

### Avenues for Improvement 

I think there are two main improvement ideas that immediately jumped out at me when considering how to improve the model. The first improvements would be to create my feature vectors based on smaller tokens in the future. This could allow for more identifiable patterns and relationships, potentially improving model performance. The second idea for improvement would be to adjust which sklearn model I used for training. I found that LogisticRegression ran very slowly and maxed out my CPU capabilities multiple times. I attempted adjusted a few model parameters, such as the solver and the max_iter, but did not see an overall improvement. It is likely that LogisticRegression was simply not the ideal classifier type to use in this scenario another and another classifier such as RandomForest would have performed much better.
