## Project description
_[Rubric](https://parsertongue.org/courses/snlp-2/assignments/rubrics/course-project/final/#reproducibility)_


Infrastructure as Code (IaC) has revolutionized the provisioning and management of cloud service models, including Infrastructure-as-a-Service (IaaS), Platform-as-a-Service (PaaS), and Software-as-a-Service (SaaS) (Kumara et al., 2021). This paradigm shift has significantly reduced organizational deployment times, from two days to just 21 minutes (Rahman & Parnin, 2023). By 2022, multi-cloud models were projected to constitute 75% of the cloud market, with investments exceeding $100 billion by 2023 (de Carvalho & Patricia Favacho de Araujo, 2020).

Despite these advancements, IaC scripts often introduce security "smells" misconfigurations that can lead to severe vulnerabilities. Common issues include insecure coding patterns (ICPs), hard-coded passwords, sensitive comments, and resource misconfigurations (Kumara et al., 2021; Rahman et al., 2019). Prior research analysis of 293 repositories revealed 21,201 instances of such smells, including 1,326 cases of hard-coded passwords (Rahman et al., 2019).

Previous research in IaC has primarily leveraged machine learning and rule-based scanners to identify and categorize these security smells. However, past efforts have largely focused on quantifying and reporting issues without providing remediation strategies. This project seeks to bridge this gap by fine-tuning meta-llama/Llama-3.2-3B-Instruct, to generate secure and efficient Terraform scripts that adhere to best practices. Our approach involves constructing a high-quality, labeled dataset from provider documentation through prompt engineering (Sahoo et al., 2024) and in-context learning (Dong et al., 2024). We will extract real-world IaC scenarios, generate initial configurations, and validate the dataset using interlabeler agreement Cohen's Kappa as a validation metric.

The fine-tuning process throught efficient parameter Low-rank adaptation (Hu et al., 2021) will be evaluated by comparing the performance of the fine-tuned Llama 3.2 model against the untuned version. We will employ BLEU (Papineni, 2002) and ROUGE (Lin, 2004) metrics to compare generated outputs against 100 input:output ground truth dataset samples from our fine-tuning data. As well as CODEBLEU, a metric for code generation, is essential for accurately evaluating code quality (Paul et al., 2024). 

**Data Collection Summary**: The data for this project is sourced from a curated list of Terraform provider documents available at [https://github.com/antonbabenko/terraform-docs-as-pdf](https://github.com/antonbabenko/terraform-docs-as-pdf). This repository contains comprehensive documentation for a total of 151 Terraform providers, covering various cloud services and configuration scenarios. The collected documents outline best practices, security guidelines, and configuration details necessary for building Infrastructure as Code (IaC) scripts.

#### My approach

Developed a script for the extraction and processing of Terraform provider documents to generate high quality Infrastructure as Code (IaC) scripts using Llama 3.2. Our approach uses the `PyPDFLoader` from `langchain_community.document_loaders` to load the 151 Provider PDFs containing their best configuration practices. This comprehensive data source provides the necessary context for the language model to successfully learn Infrastructure as Code scripting.

After processing the documentation of a single provider, we develop clear and refined instructions along with example templates to guide the model in structuring its output. This structured guidance ensures the model's output aligns with the data fine-tuning requirements we will encounter later on.

#### Instruction
```bash
1. Generate as many meaningful prompts based on the provided content.
2. Ensure each 'prompt:completion' pair is consistent in the following format:
   - **prompt**: 'Your generated prompt here'
   - **completion**: 'Your detailed, well-structured answer here, including ready-to-use IaC code snippets.'
3. Include ready-to-use IaC code in the 'completion' that adheres to best configuration practices.

# Example

- **prompt**: 'A prompt based on the content'  
  **completion**: 'Generated content'

- **prompt**: 'A prompt based on the content'  
  **completion**: 'Generated content'

```

We then establish a base system message that frames the role of the language model as an expert in IaC development. We also include a user message that presents the model with a task: to generate as many relevant prompt-completion pairs as possible using the content from the loaded PDF. This setup ensures that the language model has a clear understanding of its role and the requirements for the generated content.

#### System Role
```bash
You are an expert Infrastructure as Code (IaC) developer and assistant. Your role is to generate secure and efficient configurations with a focus on security best practices, managing secrets securely, and creating modular, reusable code from the provided content. You excel at analyzing provided documentation to produce accurate, well-commented, and structured scripts, emphasizing critical security considerations.

Given the provided context, generate as many Infrastructure as Code (IaC) prompt:completion pairs. Each script should have well-structured comments to explain the purpose and significance of the configuration. Ensure the scripts are secure, modular, reusable, and suitable for automated workflows. Highlight necessary security controls.

Here is the content:

[Provider Documentation]

```

The script then iterates a series of questions aimed at exploring different aspects of secure and efficient IaC development. These questions guide the model in producing comprehensive and contextually relevant outputs. For instance, some questions ask the model to explain the security measures required in the code, specify input parameters and expected outputs, or describe how the code should handle errors and exceptions. Each question is then ran through our model's pipeline to generate our IaC dataset.

##### Questions used to guide our data creation:

1. As an expert code developer specializing in Infrastructure as Code (IaC), please provide example scripts. Include descriptive comments above each script section that explain the configuration, emphasizing security practices, their significance, and any best practices recommended based on the provided context.
2. Extract all of the available script argument references.
3. What are the exact requirements for this code? Specify input parameters, expected outputs, and any constraints or conditions.
4. How should the code handle errors, exceptions, or invalid input? Include any specific validation or fallback mechanisms needed.
5. What security measures are crucial for this code? List areas where data should be sanitized, access should be restricted, or secrets must be securely managed.
6. What systems or APIs does this code need to integrate with, and how should the integration be handled?
7. What level of documentation is required for this code? Should the code include inline comments, or should a README file be provided for usage instructions?

The model generates output for each question, which is then written to a text file in our desired output structured format. This method facilitates systematic analysis and review of the model's performance. Here are two examples illustrating the output when provided with the best security practices for the Docker Provider.

##### Here is an example of one of our generated outputs:

```bash
**'prompt'**: *Configure Docker provider settings*  
**'completion'**:  
provider "docker" {
  # Specify the Docker host address
  host = "tcp://127.0.0.1:2376/"

  # Specify the Docker registry authentication settings
  registry_auth {
    address = "registry.hub.docker.com"
    config_file = "${pathexpand("~/.docker/config.json")}"
  }
  
  registry_auth {
    address = "registry.my.company.com"
    config_file_content = "${var.plain_content_of_config_file}"
  }
  
  registry_auth {
    address = "quay.io:8181"
    username = "someuser"
    password = "somepass"
  }
}
```
