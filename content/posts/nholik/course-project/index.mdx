## Project description

My project aimed to do some experiments with (open) LLMs to generate synthetic data to train "classic" NLP tools.  To limit the scope, I specifically focused 
on being able to train a POS classifier using data generated purely from two LLMS: Meta's llama3.2 (3b) and Google's gemma2 (24b).  I compared these against
a basline human annotated training set (see [here](https://github.com/UniversalDependencies/UD_English-EWT/tree/master)).  Originally, it was hoped that 
I would do NER recognition too, but sometimes you have to be a little more realistic.

The reasoning behind this task was two-fold.  From a pedagogical standpoint, I wanted to explore a few areas of learning: those from the past (POS tagger from my 
first couple classes), those from the current class, specifically getting some practice working with pytorch, and of course figuring out how to mix in LLMs as 
is the way of late.

From my admittedly amateurish research, it seems that using bi-directional LSTM NN's and embeddings like BERT are the SotA methods, but in some early experimentation, 
I found that BERT with a simple CNN pretty much gave a 90% F1 score with hardly any training data, so threw that out immediately and focused on using some simpler
embeddings, namely Glove, and a CNN.  These choices helped me with working more with some things I have been exposed to and have had more time to work with to take
this more in the learning direction.  It was a useful start though, and helped me really appreciate how powerful BERT is, but it didn't seem to be what would help 
in my experimentation nor with achieving my goal of playing with more fundemental tools.

My other, perhaps practical motivation, was to run an experiment to both see how well two models, one very low powered (llama3.2 3b) and one still pretty low powered 
but much more powerful relatively speaking (gemma2 24b) would be as far as creating synthetic data, then of course seeing how well they matched up against 
a tried-and-true human annotated training set.  The idea being here, to see one, how much you can get from a lower powered model to replace human tagging, and 
seoncdarily how well in general a synthetic source might be able to provide data.  While POS tagging is clearly not a hot topic, there is interest [example](https://arxiv.org/pdf/2410.12879) in how low resource languages could benefit from transfer learning from and POS tagging still finds its way as a way to help with other tasks 
see [this](https://arxiv.org/pdf/2405.13319) for example (one could then surmize perhaps leveraging LLM -> low resource -> tools for NLP -> something useful?).  
My additiona, and perhaps naive, thoughts are, inference on LLMs takes a lot. My resultant classifier takes very little resources, and with "AI everywhere" (whatever 
that really means) having the ability to run on limited resources (mobile, low powered cloud compute, microwave -jk - or maybe not) is a viable problem.  So 
if one trains the "kitchen sink", extracting useful, let's say emergent, things to help more focused tasks seems like a win.

My results are that of a fumbling student, but I do think although not the type of work that will be getting me any attention, do seem like a viable starting point - they probably just need more experience behind them and more fleshed out ideas.

## Results

As vaugely alluded to, I had narrowed the scope here to just POS tagging and using a CNN model with Glove for classification. 
My basline data was from https://github.com/UniversalDependencies/UD_English-EWT/tree/master and my goal was to have my model perform POS 
tagging using the UD POS tags.  To generate synthetic data, I did some "prompt engineering" and came up with
```
Generate exactly 10 complete and grammatically correct sentences. Each sentence must be tokenized and tagged using only Universal Dependencies (UD) POS tags, formatted as [(word, POS), (word, POS), ..., (word, POS)]. 
The 10 sentences together must include every UD POS tag at least once, including PROPN for proper nouns. Follow these strict instructions:
Rules:

    Complete and Unique Sentences: Each sentence must be standalone, meaningful, grammatically correct, and unique. Avoid using or echoing the example sentences below.
    Include All UD POS Tags: Across the 10 sentences, include all UD POS tags, such as NOUN, VERB, DET, ADJ, PRON, ADP, ADV, AUX, PUNCT, PART, PROPN, CCONJ, SCONJ, NUM, SYM, X, and INTJ.
    Proper Nouns (PROPN): At least 3 sentences must include proper nouns (e.g., names of people, places, organizations) tagged as PROPN.
    No Extra Text: Output only the 10 lists of tokenized and tagged sentences, with no explanations, examples, or headers.
    Diverse Syntax and Vocabulary: Use a variety of sentence structures (declarative, interrogative, exclamatory, imperative, compound, active voice, passive voice, past tense, present tense, future tense) and lexically rich vocabulary.
    Subject Matter: At least 1 sentence should mirror a real-world scenario or event, such as a conversation, news headline, or social media post.
    Exact Format: Each line must represent one complete sentence, formatted as [(word, POS), (word, POS), ..., (word, POS)].

Example Format (do not reuse these sentences):

    [(The, DET), (cat, NOUN), (slept, VERB), (peacefully, ADV), (., PUNCT)]
    [(Maria, PROPN), (and, CCONJ), (John, PROPN), (traveled, VERB), (to, ADP), (Paris, PROPN), (last, ADJ), (week, NOUN), (., PUNCT)]
    [(What, PRON), (a, DET), (beautiful, ADJ), (sunset, NOUN), (!, PUNCT)] 
    [(Oh, INTJ), (, PUNCT), (I, PRON), (forgot, VERB), (my, DET), (wallet, NOUN), (!, PUNCT)]

Now, generate exactly 10 tokenized and tagged sentences that meet these criteria:
```

The data was generated by using ollama and a simple node.js script to run this prompt over and over again for a set amount of times, streaming the output for text files.
I ran the prompts long enough to get at least 10,000 samples which was on part with the human tagged data set referenced above had. There were some interesting 
observations when refining the prompt, like llama3.2 would often go into a loop of just repeating the same sentence over and over ad-infinitum (I had to break the process),
while gemma2 was much more on-task, so to speak.  Ultimately, the prompt above proved to give reliable and seemingly random data.  Perhaps the most notable thing is
that llama3.2 gave much more varied responses, while gemma2 loved to give the good old typing exercise as its first response nearly every time:

## Error analysis

My error analysis is for the most part baked into the results.  By the nature of the task, 
I wanted to compare by keeping the classifier constant the results of the various data sources.
Some things, I looked at a little closer were for instance why llama3.2 was so bad at SCONJ.  While 
not surprising this less used tag would not classify as well, it seems it was also compounded by 
just have really bad examples in the generated data
```shell

[("Can't", 'ADJ'), ('you)', '[PRON] [understand], [VBP] [that], [SCONJ], [the], [DET], [world], [NOUN], [is], [VERB], [ complicated], [ADJP], [?], [PUNCT]')]
... ('.', 'SCONJ')]
('[Yesterday]', 'SCONJ'), ('was', 'VERB'),
('industry?', 'SCONJ')
```
overall, most of the 400 examples were just oddities.

While developing the model, I ran validation metrics against the split training set on each epoch 
to get feedback.  Overall though, I was more focused on just baselining, not getting a perfect model. With 
a good enough baseline, I didn't find it was as germane to the project to do a detailed analysis.  If I was a
better student in this part, I really should have investigated why ADV did so poorly, though 
basic inuition and observing
```
 ADJ     0.8447    0.7977    0.8205      1794
 ADV     0.7908    0.7540    0.7720      1183
```
kind of lends to some evidence that classfication of adv vs adj is hard and there is likely a trend
using lingustic intuition and the presiscion/recall data that ADV has a tendency to misclassify to ADJ 
thought not the other way around. This trend holds even with the best syn-data:

```
 ADJ     0.7763    0.7681    0.7722      1794
 ADV     0.6426    0.6534    0.6479      1183
```
though it is not as conclusive here.  I could speculate (shame on me for speculating during error analysis)
is that that very same misclassification pervades in the synthetic data.

Another issue hit on earlier is with the data generation itself.  As noted, there was a lot of repeated data (not so much for llama3.2, but it was stil over 10%).  
This is something I would need to better control, but really have no good ideas.  The LLM is a black box thing is quite evident in that regard.

## Future improvements

There are definitely a lot of ways this could go and be improved:

- Using more and varied LLMs to see what trends there are.  My 2 are hardly enough to really say too much with certainty
- Tweaking the models, changing the temperature and seeing how that helps (or hinders) the random data generation
- Using existing data sets, like wikpedia articles to prime the models to help guide toward more varied data
- Using different classification models.  Thought not presented, I did an initial pass with BERT, but it was a bit too good on the human tagged data.  It may or may not be telling to see how a simpler n-gram + basic feature model, glove, BERT (others?) show how well the synthetic data does and how much of it is the model itself (this is admittidly a serious short coming in my experiment, and the hand waving yeah I did try BERT, trust me is obviously leaving more to be desired)
- Deeper dive into what the syn-data gets wrong and how it correlates to the generated data set and finding where perhaps the syn-sourced model actually does better (there must be at least one example, right?)

Overall, this is a pretty basic, but at least shows glimpses of some possible utility.  A really great enhancement, which in my overly optimistic take thought I might do, is seeing 
how this could help with a low resource or at least foreign language. I would imagine with my approach as is, it would do pretty poorly, but it might be jumping off point to see where 
it could be used to augment human data.  Perhaps it doesn't need to be all or nothing here.
