## Task summary

The end goal of this task is to be able to when provided with two snippets of text to determine if said snippets are from the same author or not.
In short this is a, seemingly simple, binary classification problem.  The main challenge is that we do not know who made the snippets.  Thus 
we are in need of a rather general classifier.  Further this differs from a straight classification problem in a few ways.  Foremost is that 
the classifier needs to be in some sense symmetric.  If I give snippetA and snippetB, I should fully expect the classifier should give the 
same exact result, positive or negative to snippetB and snippetA.  This clearly differs from the type of classifier I made back in LING-539, as
that would just take a single input to get a classification, while here I am feeding in a pair, and order should not matter.

Thus at a high level, I am tasked with taking a pair/tuple (text_1,text_2) and getting a number out 0 or 1 representing different and 
same authorship respectively.
In order to do this and with my observation that this is a bit different than other classification problems with this 'twist', 
it was clear that I need to explore how this type of problem is done in practice.  Some initial research landed me on [this book](https://link.springer.com/book/10.1007/978-1-0716-0826-5),
in particular its overview of the idea of a Siamese Neural Network:
> Similarity has always been a key aspect in computer science and statistics. Any time two element vectors are
> compared, many different similarity approaches can be used, depending on the final goal of the comparison ...
> But if the comparison has to be applied to more complex data samples, with features having different
> dimensionality and types which might need compression before processing, these measures would be
> unsuitable. In these cases, a siamese neural network may be the best choice...

So my problem in more concrete terms, is one of 'fingerprinting'.  I don't need to know who the fingerprints belong to, but I just 
need to devise a means so that ones that are similar move the model above .5 and those dissimilar below.

Of course I will need more than just this idea, and after an initial look at the data which I go into my detail in the EDA section, 
I found I will also need to supplement.

My proposed initial line of "attack" was:

- Pick some existing word embedding method (BERT?)
- Use some supplemental data like [this](https://www.kaggle.com/datasets/michaelrussell4/10000-books-and-their-genres-standardized) to augment with some features like genre, author sex, time period.
- use the supplemental data to help make input vectors to my "twin" networks
- use the supplemental data to also feed into my "twin" networks
- create the similarity network by concatenating the outputs as the input

For the most part I kept faithful to this, but in doing this, I found a final method, much simpler but more effective.  So like before,
I made some friends along the way through many false turns, but those relationships and what I learned were quite valuable, even if not
ultimately fruitful.

Comparing my approach with SOTA approaches, I can say it probably aligns with best practices and ideas to leverage pre-trained embeddings for 
semantic similarity, but I'll fully admit I hardly did a vast research review.  One idea, though not taken directly I came across was
from [this paper](https://www.nature.com/articles/s41598-022-13690-4) to use ensemble learning.  I felt like, and probably would do this more so 
if time was not a constraint, I did this in some sense, with my genre-model and splitting my "main" model into two seperate models, one a 
CNN to simulate word-level n-grams, and a sytlometric model to capture POS features, reading level (and genre as discussed).  Further, I added 
simple 'raw-similarity' model to do some approximation of ensemble learning.

So to conclude the intro,  take a data file with ID,TEXT where TEXT are two pieces of text delimited by " [SNIPPET] ", vectorize and analyze
those texts for basic features, train a model such that when a given feature vector is fed in, texts from similar authors should result in 
close by vectors, while different authors should 'repel'.  I did this, and it kind of worked, but some of the ideas explored led to one final,
and simpler approach.  So let's get into it.

## Approach

As alluded to in my earlier sections, the goal was to use some pre-trained embeddings and make a siamese NN that took
the embeddings plus some stylometric features to build a final model. For the actual model, see `classifier.py` which you
should be able to run directly after downloading my supplementary data.  Diving into more specifics:

I have the following submodels:
- a genre classifier, which served more as a jumping off point and toying with torch.  It used sbert to give the average embedding and max embedding over all sentences in some gutenberg books.
- a stylometric model.  this a simple sequential NN which took POS frequencies, token counts, unique token ratio, various reading level scores, sentence count, word size etc
- a n-gram model using BERT embdddings and a CNN to simulate (?) capturing various word level n-grams, heavily inspired by comments in the class forum
- what i called a 'basic similarity' model, which took the cosine similarity of the sbert embeddings of the snippet, and the "authorship" vector (explained below)
- the siamese model which ran each snippet through these various models, then took the concatenated result + the basic similarity result and applied a simple NN to get a 0->1 score

The authorship vector is where I started to explore some things "on the side".  You'll see why in the results section.  But forgoing that,
I had some fun taking my large gutenberg paragraph dataframe, applying sbert and getting the average sbert embedding across all paragraphs
for a given author to make a 'fingerprint' of sorts.  More details and replication can be found in the `AllGuten` notebook (it's not really 
all of it, but it was my working nomeclature).  In the notebook, you will find some nice visualizations showing authorship clusters. *This was a key 
realization.*  All along, I have been trying to fingerprint authors, well, I have a big corpus that I can reduce down to said fingerprint.

### More on author fingerprinting

Looking at visualizations of how authors cluster, and doing some sample queries, finding things like Schopenuer's embedding had top-5 
closest to philosophical writers, Jack London - other 'outdorsy' writers, Asimov - sci-fi, I thought I had really hit on something here.
So I pik'ld the vector and used it as a feature.  More in the results.  But this was the next key realization.  My wife mentioned when

search system is good, you are probably search for very similar if not the same thing*.

So here is the moment we have all been waiting for.  As I said, nothing sophisticated here, in fact, I noticed as I took out various parts 
of the model for the stylometric features, they were not adding a lot, not was the 'simple similarity' model.  It was quite disappointing, but

the closest author, the results were kind of disappointing.  I should have kept better track, but the results, for things like clear sci-fi 
or fantasy snippets gave back authors that just seemed purely random.  So anyway, no wonder.  But maybe this bad idea would work: SBERT 
does semantic search and I have a large corpus of texts by author, maybe good tool but wrong usage?  This example was the inspiration:
https://github.com/UKPLab/sentence-transformers/blob/master/examples/applications/semantic-search/semantic_search.py I hypothsized the following,
I bet I won't get many if any false positives if I take the intersection of authors doing a semantic search for snippet 1 and same for snippet 2
and say they're from the same person if they both result in the same author.  Finally, my hypothesis was right.  And the code is dead simple:

```python
import torch
from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score, confusion_matrix, classification_report

y_pred = []
y_true = []
false_pos = 0
for _, row in df_train.iterrows():
    snip_1, snip_2 = row['TEXT'].split(' [SNIPPET] ')
    
    query_embedding_1 = model.encode(snip_1, convert_to_tensor=True)
    query_embedding_2 = model.encode(snip_2, convert_to_tensor=True)
    
    similarity_scores_1 = model.similarity(query_embedding_1, corpus_embeddings)[0]
    similarity_scores_2 = model.similarity(query_embedding_2, corpus_embeddings)[0]
    
    _, indices_1 = torch.topk(similarity_scores_1, k=5)
    _, indices_2 = torch.topk(similarity_scores_2, k=5)
    
    selected_rows_1 = df.values[indices_1.cpu().numpy(),1]
    selected_rows_2 = df.values[indices_2.cpu().numpy(),1]
    common = set(selected_rows_1).intersection(set(selected_rows_2))
    pred = 1 if len(common) > 0 else 0
    actual = row['LABEL']
    
    y_true.append(actual)
    y_pred.append(pred)
    
accuracy = accuracy_score(y_true, y_pred)
precision = precision_score(y_true, y_pred)
recall = recall_score(y_true, y_pred)
f1 = f1_score(y_true, y_pred)

conf_matrix = confusion_matrix(y_true, y_pred)

class_report = classification_report(y_true, y_pred)

print(f"Accuracy: {accuracy:.2f}")
print(f"Precision: {precision:.2f}")
print(f"Recall: {recall:.2f}")
print(f"F1 Score: {f1:.2f}")
print("Confusion Matrix:")
print(conf_matrix)
print("\nClassification Report:")
print(class_report)

```

I actually got better results on this that my complicated model.  It doesn't feel right, but still within the spirit of the competition:
- Use an open model and supplementary data set
- Create embeddings for the supplementary corpus
- Given two snippets, do a similarity search in the embedding corpus, get the corresponding indices for the top-2 and see if the intersection
of authors is non empty

So really that's the model?  It really became a search problem.  Now of course this has some limitations, the search space is frozen in
