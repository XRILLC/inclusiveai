## Project description

### Overview of project

This project centers on supporting students in developing solutions to natural language processing (NLP) competitions on [Kaggle.com](https://www.kaggle.com/competitions?tagIds=13204&participationFilter=open). It uses the ongoing Kaggle competition [Contradictory, My Dear Watson](https://www.kaggle.com/competitions/contradictory-my-dear-watson) as an example. This competition involves classifying pairs of sentences into three categories: **contradiction**, **neutral**, or **entailment**. The competition spans 15 languages, challenging participants to build models that can understand relationships between sentences across diverse linguistic contexts. The competition also requires submissions to come directly from a Kaggle Notebook, making it an ideal case study for this project.

For this competition, the project provides:

- a competition-specific [starter notebook](https://www.kaggle.com/code/mikeb3665/mdw-starter-notebook)
- a competition-specific "[virtual teammate](https://chatgpt.com/g/g-675426b991e48191be1ec7f9abf4873c-contradictory-my-dear-watson)"
- a competition-agnostic [Kaggle Docker container](https://github.com/ciioprof0/kaggle)

### Description of its novelty

This project stands out by combining multiple innovative components to enhance the learning experience for students tackling Kaggle NLP competitions. While many '[Getting Started](https://www.kaggle.com/competitions?hostSegmentIdFilter=5)' competitions provide basic starter notebooks to familiarize participants with tasks and data, this project goes further by integrating a [virtual teammate](https://chatgpt.com/g/g-675426b991e48191be1ec7f9abf4873c-contradictory-my-dear-watson): an AI-powered chatbot that provides real-time guidance, answers questions, and suggests code improvements. The virtual teammate transforms passive learning into an interactive process, helping students overcome challenges and gain a deeper understanding of competition tasks.

Additionally, the project introduces a customized version of [Kaggle's Docker container](https://github.com/ciioprof0/kaggle), enabling students to replicate Kaggle's runtime environment locally. This ensures seamless code compatibility between local development setups and Kaggle.com, reducing friction and enhancing productivity. By integrating these components, the project creates a more holistic and supportive ecosystem for students, combining interactivity, accessibility, and technical reliability.

### Motivation for project

My motivations for this project stem from personal experiences as an online programming student where I encountered recurring challenges: lack of repeatable structure in applying tools, difficulty engaging with instructors and peers, and issues with code portability.

First, computer science-related courses often introduce new tools and concepts that are used briefly before moving on. Without a structured approach to applying these tools, it is easy to forget how to use them when needed later. I wanted to address this by creating resources that reinforce consistent workflows.

Second, as an online learner, the absence of synchronous interaction with instructors and classmates can be frustrating, especially when struggling with complex problems. This lack of immediate feedback inspired me to explore ways to provide on-demand support for students tackling similar challenges.

Finally, I have experienced frustration in moving code between local development environments and platforms like Kaggle.com. Compatibility issues disrupt workflows and make it harder to focus on learning and problem-solving. This frustration motivated me to find a way to bridge the gap between these environments.

These motivations directly shaped the project's focus on structure, accessibility, and interoperability.

### Challenges of Task (with Examples)

This project addresses several key challenges that arise in NLP competitions, particularly in multilingual and online learning contexts:

1. **Handling Linguistic Diversity**
   The "Contradictory, My Dear Watson" competition spans 15 languages, requiring models to capture semantic relationships across vastly different linguistic structures and vocabularies. For example, sentence pair classification in agglutinative languages like Turkish poses unique challenges compared to isolating languages like Chinese. Traditional NLP models often struggle to generalize effectively across such varied languages.

2. **Limited Training Data for Low-Resource Languages**
   While high-resource languages like English are well-represented in datasets, low-resource languages often have fewer examples, leading to imbalanced model performance. For instance, the dataset may include only a handful of sentence pairs in languages like Swahili, making it harder for the model to achieve reliable predictions in these cases.

3. **Integrating Local Development with Kaggle's Platform**
   Students often prefer developing models in local IDEs equipped with debugging tools and linters. However, transferring this work to Kaggle.com for final submissions can result in compatibility issues, such as mismatched library versions or runtime differences. For example, a student's local code might use a specific Python package that is unavailable or outdated in Kaggle's runtime environment.

These challenges highlight the need for robust, innovative solutions, such as multilingual pre-trained models, strategies for low-resource learning, and development environments that ensure seamless compatibility.

### Discussion of state-of-the-art approaches

Several state-of-the-art techniques can enhance the outcomes of this project, particularly in improving the classification task and overall usability:

1. **Transformer-Based Models for Multilingual NLP**
   Leveraging pre-trained models like **mBERT (Multilingual BERT)** or **XLM-R (Cross-lingual RoBERTa)** can significantly improve the classification accuracy for the multilingual dataset in the "Contradictory, My Dear Watson" competition. These models excel in understanding semantic relationships across diverse languages, addressing the challenge of linguistic variability inherent in the task.

2. **Zero-Shot Learning with Large Language Models (LLMs)**
   Employing advanced LLMs, such as **OpenAI's GPT-4** or **LLama3**, can facilitate zero-shot or few-shot learning to handle low-resource languages in the competition. These models can generalize well across unseen data, making them ideal for sentence pair classification in languages with limited training examples.

3. **Interactive Learning Environments**
   Integrating an **AI-powered tutoring system** based on reinforcement learning (e.g., fine-tuned conversational agents) can improve the "virtual teammate" feature. Such systems can adapt dynamically to a student's progress, offering personalized suggestions, explanations, and alternative solutions as they navigate the competition.

These approaches align with the project's goals of providing robust technical solutions while fostering an engaging and supportive learning environment for students.

## Error analysis

Error analysis of the above results reveals insights into the model's performance across different languages. The analysis focuses on identifying patterns in misclassifications and highlighting areas for improvement.

1. **General Trends**:
   - The matrices show that **entailment** and **contradiction** are classified more accurately than **neutral** across both iterations.
   - Misclassifications are more frequent between **entailment** and **neutral** than between **contradiction** and the other classes.

2. **Performance Changes**:
   - The second confusion matrix reveals a slight decrease in accuracy for **entailment** (from 88% to 86%) and **neutral** (from 84% to 82%), while **contradiction** remains constant (87%).
   - This suggests a minor trade-off in the performance of certain categories, influenced by hyperparameter adjustments or feature selection.

3. **Consistent High Accuracy in Certain Languages**:
   - Languages like **Bulgarian**, **Russian**, **Vietnamese**, and **Thai** maintain high accuracies (96%-99%), demonstrating robust model performance in these linguistic contexts.
   - High-resource languages, such as **French** and **German**, also exhibit consistent performance, likely benefiting from well-represented training data.

4. **Lower Accuracy in Specific Languages**:
   - **English** consistently underperforms (76%-79%), despite being a high-resource language. This anomaly could indicate:
     - More complex sentence pair relationships in the English dataset.
     - Higher diversity in sentence structure or semantics compared to other languages.
   - Similarly, **Swahili** and **Urdu** have occasional dips in accuracy (94%-95%), potentially due to limited representation in the training dataset.

## Future improvements

### Discussion of Limitations

While this project provides support for students participating in NLP competitions, it has several limitations. First, the "virtual teammate" relies on predefined responses and code suggestions based on the context of the competition. This approach, while helpful, cannot fully adapt to complex, nuanced problems or offer deeper insights beyond its training. For instance, if a student deviates from the starter notebook's workflow or experiments with unconventional approaches, the virtual teammate may not provide relevant guidance.

Additionally, the customized Kaggle Docker container addresses compatibility issues but is limited to replicating Kaggle's runtime environment at a specific point in time. Changes in Kaggle's platform or the introduction of new features may require frequent updates to the container. Without ongoing maintenance, the environment could fall out of sync with Kaggle.com, creating friction for users.

### Avenues for improvements

Two enhancements could significantly increase the utility and accessibility of this project. First, expanding the capabilities of the virtual teammate by integrating a fine-tuned large language model (LLM) could improve its adaptability and contextual understanding. By leveraging more advanced models, the virtual teammate could provide more robust explanations, personalized feedback, and suggestions tailored to a wider range of workflows.

To address the limitations of the Docker container, implementing a continuous integration (CI) pipeline to monitor and update the container based on changes to Kaggle's runtime environment could ensure consistency. Additionally, allowing users to customize the container to suit their specific projects while maintaining a base template would enhance its flexibility.
